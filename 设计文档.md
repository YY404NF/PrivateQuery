# 设计文档

论文标题：基于分布式点函数的双服务器私有查询系统设计与实现  
英文标题：Design and Implementation of a Two-Server Private Query System Based on Distributed Point Functions

## 要求

1. 如果没有经过明确许可，请不要修改本设计文档。
2. 使用当前稳定的技术栈与工具链，代码尽量采用现代语法，避免为了兼容旧版本引入额外复杂度。
3. 前后端都需要有基本的模块划分，体现完整系统设计，但不做过度工程化拆分。
4. 本项目面向本科毕业设计，在保持正式学术表达的同时，控制技术深度和实现复杂度。

## 一、文档定位

本文档用于完成系统的顶层设计，明确系统的应用场景、总体架构、模块边界、核心交互流程与后续实现范围。  
当前设计以“能够完成演示、能够支撑论文撰写、能够体现双服务器私有查询核心思想”为目标，不追求工业级大规模部署，而强调结构清晰、流程完整、实验可复现。

## 二、设计目标与范围

### 2.1 设计目标

1. 设计一个基于分布式点函数（Distributed Point Function, DPF）的双服务器私有查询系统。
2. 在不暴露用户真实查询目标的前提下，实现对结构化数据记录的私有检索。
3. 提供一个可视化前端，用于展示查询输入、双服务器请求、响应重构和最终结果。
4. 提供一个轻量但完整的前后端系统，既能体现密码协议思想，也便于本科毕业设计实现。

### 2.2 设计范围

本系统重点覆盖以下内容：

1. 客户端根据目标记录生成查询密钥份额。
2. 两个后端服务器分别接收密钥份额并对本地数据执行求值。
3. 客户端对两个服务器返回结果进行重构，得到最终查询结果。
4. 系统提供演示界面，展示私有查询过程中的关键数据流。

本系统暂不重点覆盖以下内容：

1. 恶意服务器模型下的复杂安全增强机制。
2. 模糊查询、范围查询、多关键词联合查询等高级检索能力。
3. 大规模分布式部署、负载均衡、集群容错等工程化能力。
4. 完整后台管理平台与复杂权限系统。

## 三、应用场景设计

### 3.1 场景设定

本系统场景设定为：**面向隐私敏感商品信息的双服务器私有查询**。

这里的“隐私敏感商品”主要指用户不希望平台直接感知其兴趣偏好的商品类型。文档中不再继续细化具体商品目录，而是将其统一抽象为具有隐私属性的商品记录集合。  
在该场景下，用户希望从目录中查询某一条商品记录的摘要信息，但不希望任一服务器单独获知其实际查询目标。

系统返回的不是单一价格字段，而是一条商品记录的摘要信息，主要包括：

1. 商品名称
2. 商品类别
3. 参考价格
4. 库存状态
5. 商家信息
6. 更新时间

该场景具有以下特点：

1. 隐私动机明确，能够自然引出“隐藏用户真实查询目标”的系统需求。
2. 数据结构清晰，便于构建基于索引的私有查询原型。
3. 查询结果具有一定业务完整性，适合在论文中描述系统输入、输出与展示界面。
4. 场景复杂度适中，既强于单一价格查询，又不会超出本科毕业设计实现能力。

### 3.2 场景抽象

在本设计中，不强调真实电商业务闭环，而是将其抽象为一个“结构化记录私有查询系统”。系统中的每条记录可以理解为一条商品信息摘要，前端首先向用户展示公开商品列表，用户点击某个商品后进入详情查看流程，但系统保护的正是这次“进入详情”的查询行为，使任一服务器都无法单独获知用户究竟查看了哪一条商品信息。

因此，本系统的业务重点不是交易流程，而是：

1. 如何组织可查询的数据集。
2. 如何将用户点击的目标商品映射到查询索引。
3. 如何通过双服务器协同完成隐私保护下的详情检索。
4. 如何同时提供流程展示视角和场景展示视角。

### 3.3 场景边界

本系统采用**面向单条记录的精确索引查询**。  
也就是说，用户最终查询的是某一条确定记录，而不是全文搜索或模糊匹配。

这样设计有两个作用：

1. 与 DPF 擅长的点查询模式保持一致。
2. 避免在本科阶段引入复杂检索引擎、全文索引与多轮交互逻辑。

## 四、系统总体架构

### 4.1 架构概述

系统采用“四角色”顶层架构：

1. 查询客户端（前端页面）
2. 服务器 A
3. 服务器 B
4. 数据准备与管理模块

其中，前端既承担用户交互功能，也承担客户端私有查询逻辑；两个后端服务器部署相同数据副本，彼此独立完成求值；数据管理模块负责数据导入、预处理和版本同步。

### 4.2 架构说明

#### 查询客户端

查询客户端负责：

1. 展示数据列表或目标记录选择入口。
2. 将用户选中的目标记录映射为内部查询索引。
3. 生成两个查询密钥份额。
4. 分别向服务器 A 和服务器 B 发起请求。
5. 对两个服务器返回的结果进行重构。
6. 在界面中展示查询结果及交互过程。

#### 服务器 A / 服务器 B

两个服务器的职责保持一致，均负责：

1. 接收来自客户端的查询密钥份额。
2. 基于本地相同的数据副本执行求值。
3. 返回本地计算结果份额。
4. 维护基础日志、健康检查与数据版本信息。

在设计假设中，两台服务器**不串通**，且均属于“诚实但好奇”模型，即会按协议执行操作，但可能尝试从本地接收到的数据中推测用户意图。

#### 数据准备与管理模块

数据准备与管理模块负责：

1. 导入初始商品数据集。
2. 对数据记录进行规范化处理与编号。
3. 将数据同步到服务器 A 与服务器 B 的本地数据库。
4. 提供数据版本号，供客户端确认当前查询的数据快照。

该模块采用轻量实现方式，不单独开发复杂后台页面。

### 4.3 逻辑结构图

系统逻辑结构可抽象为如下关系：

```text
┌──────────────────────────────────────────────┐
│                查询客户端（前端）            │
│  商品列表展示 | 目标映射 | DPF密钥生成        │
│  双路请求发送 | 结果重构 | 流程/场景展示      │
└───────────────┬───────────────────┬──────────┘
                │                   │
                │ 请求份额A          │ 请求份额B
                │                   │
        ┌───────▼───────┐   ┌───────▼───────┐
        │   服务器 A     │   │   服务器 B     │
        │ 接口层         │   │ 接口层         │
        │ 求值层         │   │ 求值层         │
        │ 数据访问层     │   │ 数据访问层     │
        └───────┬───────┘   └───────┬───────┘
                │                   │
                │ 本地数据副本       │ 本地数据副本
                │                   │
        ┌───────▼───────┐   ┌───────▼───────┐
        │ SQLite A      │   │ SQLite B      │
        └───────────────┘   └───────────────┘
```

该结构图体现了三个关键点：

1. 客户端是私有查询逻辑的发起者和结果重构者。
2. 两个服务器彼此独立，分别处理不同的查询份额。
3. 两个服务器各自访问本地相同数据副本，而不是共享同一个数据库。

### 4.4 部署结构

本系统采用本地三进程加双数据副本的轻量部署方式：

1. 一个前端开发服务器或静态前端页面实例。
2. 一个服务器 A 进程，对应一份本地 SQLite 数据文件。
3. 一个服务器 B 进程，对应一份本地 SQLite 数据文件。

部署时由两个后端服务在启动阶段自动完成样例数据加载和数据库初始化。  
这种部署方式既符合双服务器模型的基本要求，也便于在本科毕业设计环境中快速演示和复现实验结果。

## 五、系统功能设计

### 5.1 前端功能模块

前端划分为以下模块：

1. 查询输入模块  
用于加载公开记录列表，支持用户点选目标记录，并展示当前数据版本。

2. 私有查询生成模块  
负责根据目标索引生成查询密钥份额，并准备发送到双服务器的请求体。

3. 结果重构模块  
负责接收两个服务器的响应并完成结果恢复。

4. 过程展示模块  
以可视化方式展示“目标索引生成 -> 双路请求发送 -> 双服务器响应 -> 客户端重构结果”的完整链路。

5. 结果展示模块  
展示最终恢复出的记录摘要信息，并提示查询状态、耗时、异常信息等。

### 5.2 后端功能模块

后端划分为以下模块：

1. 接口层  
负责参数接收、请求校验、响应返回。

2. 查询求值层  
负责执行双服务器私有查询中的本地求值逻辑，是后端核心模块。

3. 数据访问层  
负责读取本地 SQLite 数据库中的记录数据或索引化数据结构。

4. 数据管理层  
负责数据初始化、版本维护、同步校验与必要的导入功能。

5. 系统支撑层  
负责日志、配置、错误处理、健康检查等通用功能。

### 5.3 前后端模块对应关系

为了便于后续实现拆分，前后端模块之间的主要对应关系如下：

1. 前端查询输入模块  
对应后端 `GET /api/catalog/list` 与 `GET /api/catalog/version` 接口，用于获取公开商品列表和数据版本。

2. 前端私有查询生成模块  
不依赖额外后端生成逻辑，由前端本地完成目标索引映射与查询份额构造。

3. 前端过程展示模块  
对应后端 `POST /api/query/eval` 接口的请求报文与响应报文，用于展示双路通信过程。

4. 前端结果重构模块  
接收来自服务器 A 和服务器 B 的结果份额，并在前端本地恢复最终记录摘要。

5. 后端数据管理层  
负责在服务启动时完成样例数据初始化，并为前端展示页提供一致的数据基础。

## 六、数据组织设计

### 6.1 数据模型

为兼顾场景完整性与实现难度，每条记录保留如下字段：

1. `record_id`：记录编号
2. `item_name`：商品名称
3. `category`：商品类别
4. `price`：参考价格
5. `stock_status`：库存状态
6. `merchant`：商家名称
7. `updated_at`：更新时间

这一设计比单独保留“价格”字段更适合写入论文，也更适合做界面展示。

### 6.2 数据组织思路

从系统实现角度，业务记录表是面向展示的数据结构；从私有查询角度，系统还需要将记录映射为可按索引访问的查询单元。  
系统采用“**展示记录 + 查询索引映射**”的设计方式：

1. 管理模块对原始数据进行编号。
2. 客户端基于编号生成私有查询目标。
3. 两个服务器对相同编号的数据副本进行本地求值。
4. 客户端将返回结果恢复为可展示的记录内容。

这种设计可以在不明显增加实现难度的前提下，把“业务展示”和“密码查询”两个层面区分开来。

### 6.3 数据规模

作为本科毕业设计，数据规模控制在几十条到几百条记录之间。  
这样既可以体现系统查询过程，又不会让数据预处理、界面渲染和实验描述过于复杂。

## 七、交互流程设计

### 7.1 主流程

系统主流程如下：

1. 客户端加载当前数据版本与公开记录列表。
2. 用户在前端选择目标记录。
3. 客户端将目标记录转换为内部索引。
4. 客户端生成两个查询密钥份额。
5. 客户端分别向服务器 A 和服务器 B 发送查询请求。
6. 两个服务器基于本地副本独立完成求值。
7. 两个服务器分别返回结果份额。
8. 客户端接收并重构结果。
9. 前端展示最终记录内容与查询过程信息。

### 7.2 展示重点

前端展示分为两个界面，分别承担不同的演示目标。

#### 流程展示界面

流程展示界面用于面向评委老师完整展示系统工作过程，应当完整展示以下信息：

1. 用户选择的目标记录编号或名称
2. 发送到两个服务器的两份不同请求
3. 两个服务器返回的结果份额
4. 客户端重构得到的最终结果
5. 本次查询的耗时与状态

该界面直接展示完整请求报文和完整响应报文，不通过展开按钮隐藏关键数据，以保证演示路径直观、连续。

#### 场景展示界面

场景展示界面用于还原实际使用情境，应当具备以下特征：

1. 首页展示公开商品列表，并显示完整商品名称。
2. 用户点击某条商品后进入详情查看流程。
3. 页面以简洁方式展示商品详情结果，而不完整暴露底层报文内容。
4. 页面不依赖额外展开按钮完成主要信息查看，以减少多余点击操作。

该界面的重点是让评委直观看到：用户执行的是一次普通的“点击进入商品详情”操作，而系统真正保护的是这次详情查询目标本身。

### 7.3 关键时序

系统关键时序可概括如下：

```text
用户          前端客户端         服务器A         服务器B
 |                |                |                |
 | 打开商品列表   |                |                |
 |--------------->|                |                |
 |                | 获取公开列表   |                |
 |                |--------------->|                |
 |                |<---------------|                |
 | 选择商品       |                |                |
 |--------------->|                |                |
 |                | 目标映射/生成份额A             |
 |                | 目标映射/生成份额B             |
 |                |--------------->|                |
 |                |-------------------------------->|
 |                |                | 本地求值       |
 |                |                |--------------  |
 |                |                | 返回结果份额A  |
 |                |<---------------|                |
 |                |                |                | 本地求值
 |                |                |                |-----------
 |                |                |                | 返回结果份额B
 |                |<--------------------------------|
 |                | 本地重构最终结果                |
 |                | 展示详情/展示流程               |
 |<---------------|                |                |
```

该时序强调两点：

1. 查询目标的处理起点和结果恢复终点都在客户端。
2. 两个服务器只看到各自收到的查询份额和本地求值过程，无法单独恢复用户的真实查询目标。

### 7.4 异常流程

为保证演示稳定性，系统在顶层设计中考虑以下异常流程：

1. 当服务器 A 或服务器 B 中任意一方不可用时，客户端终止本次结果重构，并在界面中明确提示“查询未完成”。
2. 当两个服务器返回的数据版本不一致时，客户端拒绝执行结果重构，并提示数据副本需要重新同步。
3. 当前端请求超时或网络失败时，流程展示页应保留已发送报文和失败状态，便于演示异常处理过程。
4. 当客户端重构结果为空或格式异常时，界面应显示错误状态，而不是直接展示空白详情页面。

## 八、技术选型

### 8.1 后端

后端技术栈如下：

1. Go  
用于实现服务端主体逻辑与私有查询求值流程。

2. Gin  
用于快速构建 REST 风格接口，适合轻量项目开发。

3. SQLite  
用于保存结构化数据集、版本信息和必要配置，部署简单，适合本科项目。

### 8.2 前端

前端技术栈如下：

1. Vue  
用于构建交互界面。

2. Vite  
用于前端工程构建与开发调试。

3. Vue Router  
用于页面级结构组织。

4. Axios  
用于前后端 HTTP 交互。

### 8.3 前端 HTTP 交互

前端 HTTP 交互统一采用 Axios，原因如下：

1. 接口封装清晰，适合建立统一的 `api` 层。
2. 支持超时、拦截器、统一错误处理，便于处理双服务器并发请求。
3. 在本科项目中更容易说明接口组织方式，也更容易调试。
4. 与 Vue 项目结合足够成熟，不会引入额外学习成本。

Axios 作为统一接口层组件写入本系统设计，并用于后续前端实现。

## 九、接口与页面顶层设计

### 9.1 接口设计

系统保留以下核心接口：

1. `GET /api/catalog/version`  
用于获取当前数据版本与基础说明。

2. `GET /api/catalog/list`  
用于获取公开记录列表或可选目标集合。

3. `POST /api/query/eval`  
用于向单个服务器发送查询密钥份额并返回本地结果份额。

4. `GET /api/health`  
用于健康检查和演示环境状态确认。

样例数据由服务启动时自动加载，现阶段不扩展复杂管理接口。

### 9.2 页面设计

前端保持 3 个核心页面：

1. 首页 / 系统介绍页  
展示课题背景、系统简介、双服务器私有查询流程简图。

2. 流程展示页  
用于详细展示客户端与双服务器交互的全部关键数据，完整呈现目标映射、密钥生成、双路请求、双路响应和结果重构过程。

3. 场景展示页  
用于展示公开商品列表与商品详情查看流程。该页面显示完整商品名称，用户点击商品后进入详情展示，系统在该过程中完成私有查询。

### 9.3 页面职责划分

各页面的职责边界如下：

1. 首页 / 系统介绍页  
负责解释课题背景、系统目标、双服务器模型和核心流程，不承担查询执行逻辑。

2. 流程展示页  
负责面向答辩演示展示协议执行过程，是报文展示、状态跟踪和结果重构的主要页面。

3. 场景展示页  
负责面向使用场景展示“商品列表 -> 商品详情”的业务流程，是最贴近真实用户操作的页面。

这种划分可以避免一个页面同时承担“讲原理”和“讲场景”两类任务，从而提高展示的清晰度。

## 十、非功能设计

### 10.1 可用性

系统应支持本地或局域网环境稳定运行，能够完成连续多次查询演示。

### 10.2 可维护性

前后端应采用基础模块划分，避免所有逻辑堆叠在单一文件中，便于后续论文整理与系统答辩说明。

### 10.3 可演示性

系统界面应尽量直观，能够让评审快速看懂“双服务器私有查询”这一核心机制，而不是停留在命令行级别。

### 10.4 安全假设

本系统默认双服务器不串通，并基于该前提讨论隐私保护效果。  
该假设应在论文和设计文档中明确写出，避免被误解为抵抗所有现实攻击模型。

### 10.5 数据一致性

系统要求服务器 A 与服务器 B 在查询阶段持有相同版本的数据副本。  
为此，服务启动时应完成样例数据初始化，并记录当前数据版本号。前端在执行查询前先获取版本信息，当发现两个服务器的数据版本不一致时，不进入正式查询流程。

### 10.6 演示稳定性

考虑到系统主要面向毕业设计答辩与实验演示，顶层设计强调以下稳定性要求：

1. 服务启动后能够自动进入可演示状态，不依赖额外人工导入数据。
2. 页面在单次刷新后能够重新加载公开商品列表和服务状态。
3. 流程展示页能够保留本次查询的关键状态信息，避免演示时出现“看不到过程”的情况。

## 十一、实现边界

为避免项目范围失控，实现边界控制在以下范围内：

1. 只支持单条记录精确查询。
2. 只支持单轮查询与单次结果重构。
3. 只做双服务器独立部署，不做集群化扩展。
4. 只实现轻量数据导入与展示，不实现复杂后台管理。
5. 重点完成“可运行、可展示、可说明”的完整闭环。

### 11.1 不纳入本次实现的内容

以下内容不纳入当前毕业设计实现范围：

1. 用户注册、登录、权限控制等账号体系。
2. 商品发布、编辑、审核、删除等后台管理功能。
3. 范围查询、多关键词查询、排序过滤等复杂检索能力。
4. 面向恶意服务器模型的强化校验机制。
5. 跨主机部署、容器编排、负载均衡与高可用架构。

### 11.2 实现优先级

后续实现阶段按照以下优先级推进：

1. 优先完成双服务器私有查询主链路，即目标映射、份额生成、双路求值和结果重构。
2. 在主链路跑通后，完成流程展示页与场景展示页两类界面。
3. 最后补充健康检查、异常提示、数据版本校验和样例数据自动加载等支撑能力。

## 十二、已确定的设计结论

当前顶层设计已经确定如下：

1. 系统场景采用隐私动机更强的敏感信息查询型，具体落在隐私敏感商品信息查询场景。
2. 查询结果返回单条记录摘要，而不是单一字段值。
3. 用户通过前端公开列表点选目标记录。
4. 前端直接承担客户端职责，包括目标映射、密钥生成、双路请求发送和结果重构。
5. 前端 HTTP 层统一采用 Axios。
6. 系统不建设独立的数据管理后台，只保留轻量数据导入与初始化能力。
7. 两个服务器各自维护一份相同的 SQLite 数据副本。
8. 当前阶段以完成顶层设计和统一关键设计口径为主，不将部署扩展能力作为主任务。
9. 商品类型保持抽象描述，不继续细化具体分类目录。
10. 公开列表显示完整商品名称。
11. 前端同时提供流程展示页和场景展示页。
12. 流程展示页完整展示请求与响应数据包。
13. 场景展示页采用简洁展示方式，不通过展开按钮承载主要信息。
14. 样例数据由服务启动时自动加载。

## 十三、当前定稿口径

如果以后需要根据本设计文档继续展开实现与论文写作，系统统一按照以下口径描述：

1. 本系统是一个基于分布式点函数的双服务器私有查询系统。
2. 系统面向隐私敏感商品信息查询场景，重点保护用户的真实查询目标。
3. 用户从公开列表中选择目标记录，客户端在前端页面中完成私有查询生成与结果重构。
4. 两个后端服务器相互独立、各自持有相同数据副本，并分别返回本地结果份额。
5. 前端包含流程展示页和场景展示页两个核心演示界面，前者展示完整报文流程，后者展示真实场景下的商品详情查看过程。
6. 系统查询对象为单条记录摘要，强调“可运行、可展示、可说明”的完整闭环。

## 十四、展示与初始化结论

1. 商品类型保持抽象描述，不在设计文档中继续细化具体目录。
2. 场景展示页中的公开商品列表显示完整商品名称。
3. 流程展示页完整展示客户端与双服务器之间的请求和响应数据包。
4. 场景展示页仅展示简略且直观的结果信息，不使用展开按钮作为主要交互方式。
5. 服务启动时自动加载样例数据，保证演示环境开箱即用。
