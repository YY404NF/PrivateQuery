列表1 本文主要工作概览

1. 构建双服务器私有查询应用场景。本文围绕公开商品列表与私密详情查询这一具体场景展开研究，将用户正常的商品浏览行为与隐私保护需求结合起来，明确系统需要隐藏的是用户真实查询目标，而不是公开展示的目录信息。
2. 完成共享DPF核心的可移植改造。本文在开源实现基础上提取分布式点函数的最小CPU实现集合，保留密钥生成、顺序求值、并行求值和聚合计算等关键能力，并删除对CUDA、OpenMP、OpenSSL和AES-NI等环境的运行依赖，形成可在普通开发环境中直接运行的共享核心模块。
3. 设计前后端协同的查询流程与结果重构机制。本文构建了由前端客户端、服务器A和服务器B组成的双服务器私有查询流程，使前端负责目标映射、份额生成、双路请求发送和结果重构，后端负责本地数据副本上的独立聚合求值，从而保证任一单侧服务器都不能独立恢复用户真实查询目标。
4. 完成系统测试与可视化演示。本文实现了首页介绍、演示页商品列表和协议时间线等界面，并结合功能测试、正确性测试和异常路径测试，验证系统能够稳定完成查询执行、结果重构和错误提示，为系统展示与论文写作提供支撑。

列表1-1 本文研究内容与研究难点

1. 研究内容
1.1 系统场景建模。本文将双服务器私有查询机制落到公开商品列表与私密详情查询场景中，完成从业务对象到查询索引的映射设计，使抽象的密码学问题能够在具体系统中得到表达。
1.2 DPF核心实现。本文围绕分布式点函数的实际应用需求，完成共享核心的裁剪与改造，实现点查询密钥生成、单点求值、全域求值与聚合计算等能力，为系统后端计算提供统一算法基础。
1.3 前后端协同演示。本文基于前后端分离架构实现了查询请求生成、双路独立求值、客户端结果重构和协议过程展示，使系统既能正确执行私有查询，也能直观展现其执行流程。

2. 研究难点
2.1 查询目标隐藏。系统需要在保证用户能够正常查看详情结果的同时，使单个服务器无法根据收到的查询份额判断真实目标记录，这对密钥份额生成与结果重构提出了较高要求。
2.2 共享核心可移植化。原始实现中存在较多与高性能环境相关的依赖，如何保留DPF核心逻辑并将其改造为适合普通开发环境运行的共享模块，是实现过程中的重要难点。
2.3 双服务器流程可视化。私有查询协议本身较为抽象，如何把份额生成、双路请求、双路响应和结果重构过程以清晰直观的方式展示给用户和评审，是系统设计中的另一关键问题。

公式2-1 点函数定义

f_(α,β)(x) =
β, x = α
0, x ≠ α

公式说明：
其中，α表示目标索引，β表示目标权重，x表示输入位置。在本文系统中，β固定取1，因此目标位置在重构后输出为1，其余位置输出为0。

公式2-2 定义域比特长度计算

in_bits = ceil(log2(record_count))
domain_size = 2^(in_bits)

公式说明：
其中，record_count表示当前记录总数，in_bits表示用于表示定义域所需的最小比特数，domain_size表示向上扩展到2的幂后的定义域大小。系统采用这种方式是为了适配DPF二叉树结构的逐层展开与求值过程。

列表2-1 双服务器模型的安全假设

1. 服务器A与服务器B不串通。两台服务器在协议执行过程中不会共享查询份额、中间结果或用户请求上下文信息。
2. 双方持有相同版本的数据副本。服务器A和服务器B在查询阶段必须基于一致的数据内容执行本地求值，否则客户端重构结果将失去正确性。
3. 服务器按协议诚实执行本地计算。系统默认服务器能够按照约定步骤完成请求解析、载荷读取、聚合求值和结果返回，但可能尝试从本地数据中推测用户意图。
4. 客户端能够同时收到两侧结果份额。只有在客户端同时获得服务器A和服务器B返回的结果份额后，才能完成最终重构并恢复目标记录详情。

列表2-2 技术选型依据

1. 后端
1.1 采用Go语言实现服务主体，原因在于其语法简洁、并发支持良好、部署方式直观，适合实现轻量级接口服务。
1.2 采用Gin框架组织HTTP接口，原因在于其路由结构清晰、请求处理简洁、便于快速搭建健康检查、目录读取和查询求值等接口。

2. 前端
2.1 采用Vue 3实现界面组件，原因在于其组件化开发方式适合组织首页、演示页、协议时间线和详情展示等页面结构。
2.2 采用Vite作为构建工具，原因在于其启动速度快、调试便捷，适合在开发过程中持续调整界面与交互细节。
2.3 采用Axios完成双路HTTP请求，原因在于其便于统一封装接口、处理超时与异常，并支持对两台服务器发起并发请求。

3. 共享核心
3.1 采用C++20实现共享DPF核心，原因在于其既适合表达底层算法逻辑，也便于通过C接口与后端集成。
3.2 采用软件AES-128 MMO实现伪随机生成器，原因在于其能够保证算法一致性，并避免将专用密码硬件依赖作为运行前提。

4. 数据存储
4.1 采用SQLite保存样例数据和元数据，原因在于其零配置、易部署、便于初始化，适合本系统的数据规模与演示环境。

列表3-1 功能需求列表

1. 公开目录加载。系统应能够向用户展示公开商品列表，使用户能够在界面中完成目标记录选择。
2. 版本一致性校验。系统应能够读取服务器A与服务器B的数据版本信息，并在正式查询前判断两侧数据副本是否一致。
3. DPF份额生成。客户端应能够根据所选目标记录生成两份查询份额，为双服务器独立求值提供输入。
4. 双路并发请求。客户端应能够同时向服务器A和服务器B发送查询请求，并接收两侧返回的结果份额。
5. 客户端结果重构。系统应能够在前端本地将两侧结果份额进行重构，并恢复为可解码的记录块数组。
6. 详情与协议展示。系统应能够同时展示最终查询结果和完整协议过程，包括生成输入、Key Share、请求体、响应体和重构结果。

列表3-2 非功能需求列表

1. 可部署性。系统应能在普通开发电脑环境中快速启动，不依赖复杂集群和专用硬件环境。
2. 可演示性。系统界面应清晰呈现双服务器结构、查询主链路和结果恢复过程，便于答辩展示。
3. 可维护性。系统应具有基本模块划分，使前后端逻辑、共享核心和数据层职责明确，便于后续修改与扩展。
4. 可扩展性。系统在保留当前精确索引查询主链路的基础上，应为后续支持更多查询类型和更强安全模型保留一定空间。
5. 稳定性。系统应能在重复查询、版本不一致和单侧请求失败等情况下保持明确状态反馈，避免出现无提示的空白结果。

列表3-3 记录载荷编码字段布局

1. record_id：8B，用于保存记录编号。
2. item_name：48B，用于保存商品名称。
3. category：24B，用于保存商品类别。
4. price_cents：8B，用于保存商品价格，以分为单位进行数值编码。
5. stock_status：16B，用于保存库存状态。
6. merchant：32B，用于保存商家名称。
7. updated_at：24B，用于保存更新时间。
8. 总长度：160B，对应20个uint64编码块。

公式3-1 固定长度载荷总长度

L = 8 + 48 + 24 + 8 + 16 + 32 + 24 = 160 Bytes

公式说明：
其中，L表示单条记录编码后的总字节长度。由于每个编码块的长度为8字节，因此160字节的记录总共可以划分为20个uint64块。

列表4-1 共享核心保留模块列表

1. Block128块结构。该结构用于统一表示DPF实现中的128位块数据，是种子、纠正字和伪随机生成结果的基础表示形式。
2. DPF密钥生成。共享核心保留根据目标索引与定义域生成两份Key Share的能力，为客户端构造私有查询请求提供支持。
3. 顺序与并行求值。共享核心同时保留顺序全域求值和并行全域求值两种能力，以支持正确性验证与后端并行聚合。
4. 聚合计算接口。共享核心保留对uint64载荷块数组执行聚合计算的接口，使后端能够直接在编码记录上执行本地求值。
5. C接口封装。共享核心对外提供稳定的C ABI接口，以便Go后端通过cgo进行调用并完成结果集成。

列表4-2 后端接口清单

1. GET /api/health
用于返回服务器名称、所属方编号、服务状态和当前数据版本，便于系统启动检查与运行状态确认。
2. GET /api/catalog/version
用于返回数据版本号、记录总数、块数和定义域大小，供前端进行版本一致性校验。
3. GET /api/catalog/list
用于返回公开商品目录信息，包括记录编号、商品名称、类别、价格文本、商家名称和更新时间，供前端展示公开列表。
4. POST /api/query/eval
用于接收单侧查询份额，完成本地载荷聚合求值，并返回该服务器对应的结果份额数组和必要执行轨迹信息。

公式4-1 结果重构公式

Result[i] = Share_A[i] + Share_B[i] mod 2^64

公式说明：
其中，Result[i]表示第i个重构后的编码块，Share_A[i]表示服务器A返回的第i个结果份额，Share_B[i]表示服务器B返回的第i个结果份额。客户端对全部20个块分别执行该运算后，即可恢复完整记录的编码结果。

列表4-3 异常处理策略

1. 版本不一致返回409。当客户端提交的datasetVersion与服务器当前数据版本不一致时，后端返回409状态码并终止查询处理。
2. 密钥格式错误返回400。当随机种子、纠正字序列或定义域参数格式不合法时，后端返回400状态码，提示请求参数存在问题。
3. 单侧服务异常提示查询失败。当服务器A或服务器B任意一侧请求失败时，前端停止结果重构并显示明确的失败提示。
4. 重构失败显示错误状态。当两侧结果份额不能正确解码或重构结果异常时，界面应转入错误状态，而不是继续展示空白详情。
5. 保留最近一次过程数据便于排查。系统在查询失败时仍保留最近一次生成输入、请求体和响应体，以便说明失败阶段并进行问题定位。

列表5-1 测试环境清单

1. 操作系统与开发环境：本地开发电脑环境，支持运行前端开发服务器、两个Go后端进程和SQLite数据库文件。
2. 前端技术栈与端口：前端采用Vue 3、Vite、Vue Router和Axios，默认通过开发服务器对外提供页面访问能力。
3. 后端技术栈与端口：后端采用Go、Gin和SQLite实现，其中服务器A默认监听8081端口，服务器B默认监听8082端口。
4. 样例数据规模与数据库文件：系统内置32条商品样例记录，分别写入server_a.db和server_b.db两个数据库文件，并在启动时自动生成数据版本号。

列表5-2 功能与正确性测试项目列表

1. 目录加载测试。验证系统启动后前端是否能够正确读取公开商品目录并正常展示列表内容。
2. 版本一致性测试。验证客户端是否能够识别服务器A和服务器B的数据版本，并在版本不一致时阻止正式查询。
3. 目标记录查询测试。验证用户选择任意一条商品后，系统是否能够完成从目标映射到详情恢复的完整流程。
4. 顺序与并行一致性测试。验证共享核心中的顺序全域求值结果与并行全域求值结果是否完全一致。
5. 结果重构测试。验证客户端对双侧返回份额执行逐块重构后，是否能够恢复正确的目标记录内容。
6. 异常路径测试。验证请求失败、参数错误和结果异常等情况下，系统是否能够给出明确提示并保留必要过程信息。
