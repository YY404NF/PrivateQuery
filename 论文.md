基于分布式点函数的双服务器私有查询系统设计与实现
Design and Implementation of a Two-Server Private Query System Based on Distributed Point Functions

摘  要

随着互联网应用的持续普及，用户在在线服务中的行为数据不断积累，查询内容、浏览对象和点击目标等信息逐渐成为具有较高敏感性的隐私数据。在传统的信息查询模式下，服务器虽然能够高效返回结果，但也往往可以直接获知用户正在访问哪一条记录、关注哪一类内容以及可能具备怎样的兴趣偏好。在商品详情查询、医疗信息检索、教育资源访问等场景中，这类目标泄露会进一步引发行为画像、偏好分析甚至隐私滥用等问题。因此，如何在保持系统可用性的同时保护用户查询目标，已经成为隐私计算与安全系统领域的重要研究问题。

针对上述问题，本文围绕双服务器私有查询机制展开研究，设计并实现了一个基于分布式点函数的双服务器私有查询演示系统。系统以公开商品列表与私密详情查询为应用场景，在保证用户能够完成正常点击与详情查看的同时，将真正敏感的目标索引隐藏在双服务器协同求值过程中。本文在整体方案上采用前后端分离结构，使用便于部署和演示的轻量技术栈完成系统实现，其中前端负责公开目录展示、查询目标映射、查询份额生成、结果重构与过程展示，后端采用统一代码库分别部署为服务器A与服务器B，并结合SQLite维护一致的数据副本。为支撑系统核心算法，本文从现有开源项目中提取并改造了可移植的DPF最小CPU实现，形成共享核心模块，去除了对CUDA、OpenMP、OpenSSL和AES-NI等环境的运行依赖，使其能够在普通开发环境下完成双服务器并行聚合求值。

在系统实现过程中，本文重点完成了共享DPF核心裁剪、结构化记录定长编码、双服务器本地聚合求值、前端交互可视化和异常处理等工作。系统将每条商品记录编码为固定长度的块数组，通过DPF求值得到目标记录对应的结果份额，再由客户端执行结果重构，从而保证任何单个服务器都不能独立恢复用户真实查询目标。为了增强展示效果，本文还设计了首页介绍、协议时间线展示和商品查询演示页面，使系统既能体现密码协议的工作原理，也能以较直观的方式支持毕业设计答辩演示。

测试结果表明，系统能够正确完成公开商品加载、目标记录选择、双路请求发送、服务器结果返回和客户端重构等完整流程。在32条样例记录的数据规模下，系统能够稳定返回正确的详情结果，并在版本不一致、请求失败和单侧服务异常等情况下给出明确提示。实验说明，本文方案在可实现性、可演示性和隐私保护表达方面具备较好的综合效果，能够较好支撑系统完整性、学术性与工程实现的综合要求。

关键词：分布式点函数；双服务器；私有查询；隐私保护；前后端分离

ABSTRACT

With the continuous growth of Internet services, user behaviors such as search targets, browsing objects and clicked records have become highly sensitive privacy data. In conventional query systems, servers can return results efficiently, but they can also directly observe which record a user is requesting. This may further reveal personal preferences and behavioral patterns. To address this problem, this paper designs and implements a two-server private query demo system based on Distributed Point Functions. The system uses a public catalog and private detail retrieval scenario, where the client generates query shares locally, two servers evaluate the shares independently on replicated databases, and the final result is reconstructed on the client side. A portable DPF core is extracted and simplified from an open-source implementation, and the runtime dependency on CUDA, OpenMP, OpenSSL and AES-NI is removed. The backend is implemented with Go, Gin and SQLite, and the frontend is implemented with Vue, Vite and Axios. Experimental results show that the system can correctly complete catalog loading, key-share generation, dual-server evaluation and result reconstruction, while keeping the user’s target hidden from any single server. The implemented system provides a complete and demonstrable prototype for undergraduate research on privacy-preserving query systems.

KEY WORDS: Distributed Point Function; Two-Server Model; Private Query; Privacy Protection; Frontend and Backend Separation

1 绪论

1.1 课题背景与研究意义

在数字化服务不断向日常生活渗透的背景下，用户越来越依赖网络平台完成信息检索、商品浏览、资源学习和服务预约等操作。表面上看，这些行为只是普通查询，但从隐私保护角度分析，用户究竟想查询什么、点开哪一条记录、关注哪一项内容，往往比最终得到的结果更具敏感性。对于电子商务平台而言，用户频繁查看某类商品详情，可能会暴露兴趣偏好和消费倾向；对于医疗服务平台而言，用户检索特定病症信息，可能映射到个人健康状况；对于教育系统和知识平台而言，用户查询某一专题内容，也可能反映学习弱项和认知需求。若服务器能够长期、完整地掌握这些目标信息，就可能形成较为精细的用户画像，从而带来隐私泄露风险。

传统查询系统的核心目标通常集中在检索效率、接口响应速度和数据组织能力等方面，而对用户查询目标本身的保护相对不足。服务器接收请求后，需要知道查询参数、过滤条件或记录编号，才能返回结果。这样一种“先暴露目标、再返回数据”的基本模式虽然简单高效，却难以满足面向隐私敏感场景的系统需求。近年来，随着隐私计算、联邦学习、多方安全计算和私有信息检索等方向的持续发展，学界逐渐开始关注如何在不暴露真实目标的前提下完成查询操作。其中，双服务器私有查询模型由于安全假设较清晰、协议结构相对简洁、实现路径较适合演示系统构建，成为隐私计算系统研究中的重要实现方向。

从研究意义上看，基于分布式点函数构建双服务器私有查询系统具有三方面价值。第一，理论价值。该系统可以将抽象的密码学机制转化为可运行的应用流程，使分布式点函数、非串通假设和结果重构等概念具备更明确的系统语境。第二，工程价值。相较于只停留在算法层面的分析，完成一个具备前端交互、后端服务、数据存储和异常处理的完整系统，更能体现方案在真实软件环境中的落地能力。第三，教学与展示价值。毕业设计不仅要体现研究思路，也要支持演示和答辩。若系统能够以可视化方式展示查询份额、双路请求和结果重构过程，则更容易让评审理解其工作机制和隐私保护意义。

本文选择“公开商品列表与私密详情查询”作为演示场景，原因在于该场景既具有直观的用户操作流程，又能自然体现“详情查看目标需要保护”的隐私动机。用户在界面中看到的是一个普通商品列表，点击后获得详情结果；但在系统内部，真正被保护的对象是目标记录索引本身。该设计既方便系统实现，也便于在论文中描述从公开信息到私有查询的层次转换。因此，围绕这一场景构建双服务器私有查询演示系统，既具备明确问题导向，也能够较好体现系统完整性、创新性和可展示性。

1.2 国内外研究现状

围绕“在不暴露查询目标的前提下获取目标数据”这一问题，相关研究经历了从理论模型到工程化原型逐步发展的过程。早期私有信息检索研究更强调通信复杂度与理论安全性，关注如何让用户在不向服务器泄露目标位置的前提下获取对应数据。随着研究深入，单服务器模型逐渐暴露出计算开销大、工程实现复杂等问题，于是引入多个服务器并假设其不串通，成为降低协议复杂度的重要思路。在这一方向上，双服务器模型由于结构清晰、推导简洁、实现难度适中而受到广泛关注。

分布式点函数是近年来在双服务器私有查询、函数秘密共享和轻量安全协议中应用较多的一类基础工具。其核心思想是把一个只在目标位置输出非零值的点函数拆分为两个函数份额，分别交给两个参与方持有。任意单方仅看到自身份额时，无法恢复目标位置；两方结果合并后，才能在目标点得到正确值。这一性质使得分布式点函数非常适合用来表达“只选中一条记录”的索引查询需求。相比直接在服务器侧暴露索引，基于DPF的方式更符合隐私保护要求；相比复杂的通用多方安全计算协议，它又能在特定场景下取得较好的实现效率。

从应用层面看，现有研究主要集中在隐私数据库访问、关键词检索、广告测量、安全统计和轻量级两方聚合等方向。很多工作会将理论协议与高性能实现结合，研究如何通过向量化、线程并行、硬件加速或专用密码库来提升求值效率。但若直接依赖CUDA、OpenMP或高性能密码硬件，虽然可以提高运行速度，却会明显增加环境部署难度，不利于在普通实验环境中稳定复现。因此，在现有研究基础上构建一个“性能适度、逻辑清楚、环境友好”的可移植演示系统，具有较强现实意义。

从系统实现角度观察，很多参考论文在整体结构上都强调需求分析、架构设计、功能模块划分、流程图展示和测试验证的完整性。通过阅读参考材料可以发现，本科毕业论文通常不追求极端复杂的理论推导，而更注重“理论基础能够说明问题、系统设计能够自圆其说、实现过程能够被验证”。这也为本文写作提供了明确思路，即在保证学术表达规范的同时，避免过度堆砌高深术语，而是结合实际工程实现，围绕系统需求、技术路线、核心机制和测试结果展开论述。

目前，面向私有查询的毕业设计型系统仍然相对较少，尤其是同时具备共享密码核心、双服务器后端、可视化前端与完整演示流程的原型更不常见。基于此，本文尝试在已有理论研究与工程经验之间建立桥梁：一方面吸收双服务器私有查询与分布式点函数的核心思想，另一方面结合轻量化Web系统的设计方式，实现一个能够运行、能够演示、能够写入毕业论文的完整原型。

2 相关技术与理论基础

2.1 私有查询与双服务器模型

私有查询的基本目标是在用户获取目标数据的同时，尽量减少服务器对真实查询意图的获知能力。与一般的访问控制、加密存储或传输保密不同，私有查询更关注访问行为本身的保密性，也就是“查了什么”这一信息不应被单个服务提供者直接掌握。在典型的单服务器场景下，即便数据库内容经过加密保护，只要服务器需要按照用户提交的目标编号或条件进行检索，查询目标就很难真正隐藏。因此，许多研究会通过引入多个不串通服务器来分散信息，使任何单一服务器都只能看到不完整的查询份额。

双服务器模型是其中较为经典也较适合教学演示的一种。其基本假设为两台服务器持有相同版本的数据副本，并且两者在协议执行过程中不会串通。客户端不直接把真实索引发送给任意一方，而是将目标信息编码成两份结构不同但能够协同使用的查询份额。服务器A只能看到份额A，服务器B只能看到份额B，两者分别在本地数据库副本上计算结果份额并返回给客户端。客户端最终对两侧返回值进行重构，得到完整查询结果。只要非串通假设成立，任意单侧服务器都不足以恢复用户真实目标。

这一模型具有明显优势。首先，它不要求复杂的多轮交互，客户端发起请求后，两个服务器可以并行完成计算，系统流程较清晰。其次，它既能体现密码协议设计思想，又可以落地为比较直观的Web系统结构，便于与前端展示结合。再次，在数据规模控制在几十条到数百条记录时，双服务器方案完全可以在普通个人电脑环境中完成演示，既不会产生过高算力压力，也能支持局域网访问和界面展示。

2.2 分布式点函数的基本原理

分布式点函数可以看作是一种针对点函数的秘密共享机制。设输入域大小为N，目标位置为α，目标输出值为β，则普通点函数在输入等于α时输出β，在其他位置输出0。分布式点函数的目标，就是将该函数拆分为两个份额，使得两个服务器分别持有一份，任意单方无法从自己的份额中识别出α，而两方输出相加后能恢复出完整点函数的值。对于私有查询系统而言，如果令β等于1，那么目标记录对应位置在重构后为1，其余位置为0，就可以把这一结果与数据库记录进行逐项乘法和求和，从而选出目标记录。

DPF的生成过程通常从两侧随机种子出发，通过伪随机生成器沿着二叉树逐层展开，结合纠正字对左右分支进行修正，使两方在目标路径上形成互补关系，在非目标位置保持抵消效果。到了求值阶段，服务器根据收到的份额，按照输入索引的二进制表示沿树向下推进，最终在叶子结点得到对应输出。若对整个定义域执行求值，则可以得到一组只在目标位置重构为1的选择向量。本文系统利用这一性质，将全域求值结果与编码后的记录块数组进行聚合，最终获得目标记录的结果份额。

对于实际系统实现而言，DPF需要同时满足三个要求。其一，密钥份额应当易于序列化，方便通过HTTP请求在前后端之间传输。其二，求值过程应当支持全域遍历和并行聚合，以避免逐条记录单独处理带来的额外开销。其三，输出群应当与记录块编码方式匹配，便于后续结果重构。基于这些要求，本文将输出群固定为模2的64次方的无符号整数加法群，使编码块、聚合结果和重构操作都能落在统一的uint64运算框架中。

2.3 双服务器模型的安全假设与适用边界

任何隐私保护系统都必须建立在明确的安全假设之上。对于本文的双服务器私有查询系统而言，最重要的前提是服务器A和服务器B在协议执行阶段不发生串通。这里的“不串通”意味着两台服务器不会交换接收到的查询份额、不会联合分析本地中间结果，也不会在协议之外共享用户请求上下文。若该假设被破坏，则两方完全有可能通过合并查询份额恢复出真实目标索引，从而使整个隐私保护机制失效。

本文还默认服务器属于诚实但好奇模型。也就是说，服务器会按照既定协议执行本地求值并返回结果，但可能试图从自身接收到的输入与计算输出中推断用户意图。这种假设与本科阶段系统实现的复杂度相匹配，可以突出“单侧不可恢复目标”的核心效果，而不必进一步引入恶意模型下的零知识证明、正确性验证和抗篡改机制。与此同时，本文系统没有处理访问模式隐藏、网络层匿名通信和跨多轮查询关联分析等更强威胁，因此其安全结论主要适用于轻量演示和基础研究验证场景。

从适用边界看，本文系统采用的是单条记录精确索引查询，不支持模糊查询、范围查询和多关键词联合检索。这一限制并非简单的功能缺失，而是与DPF在点查询场景中的表达优势直接相关。通过把系统边界控制在目标清晰、流程单轮、结果可重构的范围内，可以更有效地展示双服务器私有查询的核心思想，同时避免在本科论文中引入大量偏离主线的复杂机制。

列表2-1 双服务器模型的安全假设

1. 服务器A与服务器B不串通。两台服务器在协议执行过程中不会共享查询份额、中间结果或用户请求上下文信息。
2. 双方持有相同版本的数据副本。服务器A和服务器B在查询阶段必须基于一致的数据内容执行本地求值，否则客户端重构结果将失去正确性。
3. 服务器按协议诚实执行本地计算。系统默认服务器能够按照约定步骤完成请求解析、载荷读取、聚合求值和结果返回，但可能尝试从本地数据中推测用户意图。
4. 客户端能够同时收到两侧结果份额。只有在客户端同时获得服务器A和服务器B返回的结果份额后，才能完成最终重构并恢复目标记录详情。

3 系统需求分析与总体设计

3.1 应用场景与需求分析

本文系统以“公开商品列表与私密详情查询”为具体应用场景。系统首页和演示页可以正常展示商品名称、商家名称和价格等公开信息，用户从界面角度看到的是常见的商品浏览体验：先浏览列表，再点击“查看详情”。然而，若系统直接把商品记录编号发送给服务器进行查询，则服务器能够明确知道用户究竟关注哪一项商品。这种行为在一般电商应用中也许并不会立刻造成严重后果，但在隐私敏感商品、研究设备、健康产品或特殊兴趣内容等场景中，详情访问目标本身就可能具有较强敏感性。

因此，本系统的核心需求不是隐藏整个数据集，也不是阻止服务器返回内容，而是隐藏“用户正在查询哪一条记录”这一事实。为达成这一目标，系统需要满足以下基本要求：前端应能够在本地把目标商品映射为内部索引；该索引应通过DPF机制拆分为两份查询份额并分别发送；两个服务器必须持有相同数据副本并对同一份逻辑数据执行独立求值；客户端需要在收到两侧返回值后完成结果重构，并将其解码为可读详情。除此之外，系统还应提供完整的流程展示能力，便于在答辩场景下说明从目标选择到结果恢复的全过程。

除了核心隐私需求外，系统还需要考虑演示稳定性与可理解性。如果系统仅有命令行输出，就很难在短时间内向评审清楚说明工作内容。因此，本文在需求分析中将“可视化展示”作为重要目标之一。系统不仅要能执行私有查询，还要能展示两份密钥、双路请求、双路响应、结果重构和错误状态。这种展示需求会反向影响系统设计，例如前端需要保留最近一次查询过程的原始JSON，后端需要返回足够的执行轨迹字段，数据结构也要尽量简洁直观。

3.3 数据组织与编码设计

为了让双服务器私有查询不仅返回某一个单独字段，而是能够恢复出一条完整记录，本文将每条商品信息组织为结构化记录，并进一步编码为固定长度的块数组。系统记录字段包括记录编号、商品名称、商品类别、价格、库存状态、商家名称和更新时间。这样的设计比只返回价格更接近真实业务详情页，也更便于论文从数据模型层面说明“私有查询结果是一条完整记录摘要”。

在具体实现中，系统使用固定宽度编码方式，将一条记录编码为160字节，共20个uint64块。不同字段分配固定字节数，字符串采用UTF-8存储，超长部分截断，不足部分以零字节补齐；数值字段则按小端方式写入。采用定长编码有三点好处。其一，后端聚合时可以直接对每个块执行统一处理，避免处理可变长度字段带来的复杂逻辑。其二，客户端重构后可以按既定偏移量稳定解码记录内容。其三，固定块长度便于在论文和界面中说明数据如何从业务对象转化为密码查询对象。

系统在数据库中区分公开展示表与载荷块表。公开表用于前端展示商品列表，保留商品名称、类别、价格文本和商家名称等可读信息；载荷块表则存储按记录编号展开后的块值，用于双服务器聚合求值。元数据表额外保存数据版本、记录总数、块数和生成时间，用于客户端在发起查询前执行一致性校验。这种“展示数据”和“查询数据”相分离的设计，使系统既能维持直观界面，又能在内部保持适合私有查询的存储结构。

3.4 系统总体流程设计

在系统整体流程上，客户端是查询发起者和结果重构者，服务器A与服务器B是独立求值方。用户打开演示页后，前端首先并发读取两个服务器的版本信息与公开商品列表；如果发现数据版本不一致，则系统直接阻止正式查询。用户点击某个商品后，前端将该商品对应的记录编号作为内部索引，并结合当前定义域大小生成两份DPF查询份额。此后，前端把份额A发送给服务器A，把份额B发送给服务器B。两台服务器各自从本地数据库读取完整载荷块数组，调用共享DPF核心执行聚合求值，然后返回各自的结果份额。客户端最终将结果份额按块相加，并依照固定编码格式恢复出商品详情。

这一流程设计有两个明显特点。第一，服务器不负责结果重构，只负责本地求值。这样可以避免任一服务器单独恢复最终明文记录，更符合双服务器私有查询的职责划分。第二，客户端除了发起查询外，还承担过程数据保存和协议展示职责。系统会记录生成输入、两方Key Share、请求报文、响应报文和重构结果，以支持演示页右侧协议时间线完整展示。通过这种方式，系统将抽象的密码协议流程转化为用户可观察的交互过程，大大增强了演示效果。

4 系统关键模块设计与实现

4.1 共享DPF核心的裁剪与实现

本文系统的核心算法能力由共享模块pq-dpf-core提供。为了避免从零实现完整DPF体系带来过高开发成本，本文在开源项目基础上提取并改造了DPF的最小CPU实现集合，只保留私有查询所需的关键部分，包括点查询密钥生成、单点求值、顺序全域求值、并行全域求值以及对记录块数组的聚合计算。同时，本文保留原始开源项目的许可证归属说明，并在共享核心中补充改造说明，使其既满足学术与工程需求，也符合软件来源规范。

在具体改造上，本文首先将原实现中依赖CUDA数据类型和主机设备修饰的部分替换为标准C++20结构，例如使用自定义Block128表示128位块数据，使用std::array与std::span替代部分平台相关容器。其次，本文删除了对OpenMP并行机制的依赖，改为采用“前缀分裂加工作线程池”的方式对全域求值进行并行处理。这种方式先根据工作线程数确定分裂深度，再将定义域对应的子树任务分发到多个线程上执行，既能体现并行聚合能力，也不会因为环境中缺少OpenMP而无法运行。再次，本文使用软件AES-128 MMO实现伪随机生成器，使共享核心能够在通用CPU环境下工作，而不必依赖OpenSSL或AES-NI指令集。

共享核心的输出群被固定为uint64加法群，这一设计与本文的记录编码方案直接相关。由于每条记录最终会编码成20个uint64块，因此只要DPF求值结果在目标位置重构为1、其他位置重构为0，就可以把全域结果向量与所有记录块做乘法累加，从而得到目标记录对应的块数组。共享核心还提供了与后端对接的C接口，以便Go服务通过cgo调用生成后的原生能力。通过这一模块化设计，本文实现了“密码核心与业务系统分离”的结构目标，也为后续扩展Wasm构建接口留下了空间。

4.2 后端服务设计与实现

后端采用统一代码库实现服务器A和服务器B，两者通过启动配置区分身份。Server A默认使用端口8081、Party值0和数据库文件server_a.db，Server B默认使用端口8082、Party值1和数据库文件server_b.db。这样的设计避免了重复维护两套代码，也能够更加清晰地说明双服务器模型中的“逻辑对称、配置区分”特征。服务启动后会自动检查数据库结构、写入样例数据并生成数据版本号，确保系统进入可演示状态。

在后端模块划分上，接口层负责处理健康检查、目录读取、版本读取和私有查询求值请求；查询服务层负责版本校验、密钥份额反序列化、载荷块读取和共享核心调用；数据访问层负责SQLite表结构创建、样例数据初始化和块数组读取；载荷编码层负责业务记录到固定长度块数组的互转。这样的层次结构虽然不算复杂，但已经足以支撑论文中对模块职责的完整说明。特别是在私有查询接口中，后端不会直接返回明文记录，而是仅返回结果份额数组和必要的执行轨迹字段，从而将结果重构职责保留在客户端。

后端的私有查询接口接收数据版本号、查询编号、定义域大小和密钥份额。服务首先检查客户端版本号与本地元数据是否一致，再检查定义域大小与纠正字数量是否合法。校验通过后，后端从数据库中读取完整块数组并调用共享核心完成聚合求值。由于载荷块以记录编号和块编号顺序存储，因此可以直接构造连续内存数组传入共享核心。共享核心返回的结果份额随后被编码为十六进制字符串，最终作为JSON数组返回前端。为保证查询过程可追踪，响应中还带有工作线程数、记录数和块数等信息，便于在演示页中同步展示。

4.4 查询执行过程与结果重构

从一次完整查询来看，客户端首先根据目标商品记录编号生成两份Key Share。每份Key Share包含随机种子、纠正字序列以及定义域相关信息。服务器A和服务器B分别收到一份Key Share后，在本地对完整载荷块数组执行聚合求值。由于非目标位置在两侧重构后为0，目标位置重构后为1，因此经过按块求和后，最终得到的结果等价于目标记录本身的块数组。客户端在收到两侧结果份额后，只需逐块执行模2的64次方加法，就能恢复出原始记录载荷，再按照固定偏移量将其解码为结构化详情对象。

这种设计与直接让服务器返回明文记录相比，最大的区别在于服务器并不知道自己所聚合出的结果到底对应哪条记录。单侧服务器只看到一个看似随机的份额输入以及本地计算出的份额输出，而不会获得完整的目标信息。另一方面，客户端通过保存生成输入与重构结果，还可以把整个过程以JSON形式呈现在协议时间线中。对毕业设计答辩而言，这种“既有算法逻辑，又有可视化证据”的表达方式比单纯给出函数结果更具说服力。

公式4-1 结果重构公式

Result[i] = Share_A[i] + Share_B[i] mod 2^64

其中，Result[i]表示第i个重构后的编码块，Share_A[i]表示服务器A返回的第i个结果份额，Share_B[i]表示服务器B返回的第i个结果份额。客户端对全部20个块分别执行该运算后，即可恢复完整记录的编码结果。

4.5 异常处理与版本一致性设计

为了保证系统在演示环境中的稳定性，本文在实现过程中重点设计了版本一致性校验和异常处理机制。首先，客户端初始化时会同时请求服务器A和服务器B的版本信息，若发现两侧数据版本不同，则阻止正式查询并在界面中显示明确提示。这样做可以避免一侧使用旧数据、一侧使用新数据时导致重构结果失真。其次，后端对私有查询请求中的版本号、定义域大小、随机种子格式和纠正字数量进行校验，发现错误后分别返回400、409或500等不同状态码，以区分参数错误、版本冲突和内部异常。

另外，系统在前端状态层保留了最近一次失败信息和阶段状态。当双路请求中有任意一侧失败时，前端不会直接清空上下文，而是保留已经生成的密钥和已发送的请求内容，方便在协议时间线中展示“查询执行到哪一步失败了”。这种做法在毕业答辩中很有意义，因为它不仅体现了系统“成功路径”能跑通，也说明系统对失败路径进行了完整考虑。对于一个强调系统设计完整性的本科论文而言，异常处理并非边缘内容，而是评价系统是否成熟的重要方面。

列表4-3 异常处理策略

1. 版本不一致返回409。当客户端提交的datasetVersion与服务器当前数据版本不一致时，后端返回409状态码并终止查询处理。
2. 密钥格式错误返回400。当随机种子、纠正字序列或定义域参数格式不合法时，后端返回400状态码，提示请求参数存在问题。
3. 单侧服务异常提示查询失败。当服务器A或服务器B任意一侧请求失败时，前端停止结果重构并显示明确的失败提示。
4. 重构失败显示错误状态。当两侧结果份额不能正确解码或重构结果异常时，界面应转入错误状态，而不是继续展示空白详情。
5. 保留最近一次过程数据便于排查。系统在查询失败时仍保留最近一次生成输入、请求体和响应体，以便说明失败阶段并进行问题定位。

5 系统测试与结果分析

5.1 测试环境与测试目标

本文测试工作的主要目标包括三个方面：一是验证双服务器私有查询主链路是否正确，即能否在客户端正确恢复目标记录；二是验证系统功能是否完整，包括目录加载、版本校验、详情展开、协议时间线展示和错误提示；三是考察系统在小规模样例数据集上的运行稳定性与基本性能表现。由于本文系统定位于演示型原型，测试重点不在极端规模压测，而在于保证流程完整、结果正确和界面可展示。

在测试环境上，系统使用本地开发电脑运行前端和两个后端服务，服务器A与服务器B分别监听不同端口并持有独立SQLite数据库文件。样例数据集包含32条记录，系统会在启动时自动写入数据表并生成版本号。共享DPF核心通过原生测试程序验证密钥生成、点值恢复、顺序求值与并行求值一致性，以及载荷聚合结果与目标记录是否匹配。前端与后端则分别通过构建命令和实际页面交互进行验证。该测试环境能够较真实地覆盖毕业设计演示时的使用方式，因此具有较好的代表性。

2. 前端技术栈与端口：前端采用Vue 3、Vite、Vue Router和Axios，默认通过开发服务器对外提供页面访问能力。
3. 后端技术栈与端口：后端采用Go、Gin和SQLite实现，其中服务器A默认监听8081端口，服务器B默认监听8082端口。
4. 样例数据规模与数据库文件：系统内置32条商品样例记录，分别写入server_a.db和server_b.db两个数据库文件，并在启动时自动生成数据版本号。

5.2 功能测试与正确性验证

在功能测试方面，系统首先验证初始化阶段能否正常完成目录加载与版本同步。测试结果显示，当前端启动并访问后端后，能够正确读取商品列表和两台服务器的版本信息；在版本一致的情况下，演示页左侧可显示全部样例商品，并允许用户逐条点击查看详情。协议时间线能够依次展示生成输入、Key Share A、Key Share B、请求A、请求B、响应A、响应B和恢复结果，说明前端状态管理与界面渲染逻辑是完整可用的。

在查询正确性验证方面，本文以共享核心自带测试与端到端页面测试相结合的方式进行。共享核心测试首先验证：当输入索引等于目标α时，两侧点值求和结果为1；当输入索引不等于α时，两侧点值求和结果为0。随后验证顺序全域求值与并行全域求值结果一致，说明并行策略没有破坏算法正确性。最后，用构造好的载荷块数组测试聚合结果是否等于目标记录。端到端测试则从前端选择商品开始，检查最终详情中的记录编号、名称、价格和商家信息是否与对应样例数据一致。测试结果表明，在32条记录、20块载荷的配置下，系统能够稳定重构出目标记录。

在异常路径测试方面，当人为制造版本不一致时，前端能够及时阻止查询并给出提示；当请求参数被破坏或服务端返回错误时，界面会将阶段切换为失败，并保留当前查询过程的上下文信息。通过这些测试可以看出，系统不仅可以完成成功路径，也能在失败路径下维持较好的可解释性。

5.3 性能分析与结果讨论

由于本文系统面向的是几十条到数百条记录规模的演示场景，因此性能分析主要关注“是否足以支撑顺畅演示”而非大规模部署极限。从实际运行情况看，在32条样例记录规模下，前端点击商品后能够较快完成本地份额生成、双路请求发送和结果重构，用户基本不会感受到明显等待。服务器端通过并行聚合策略提升了全域求值阶段的处理效率，在普通个人电脑环境下即可完成稳定响应。由于样例数据规模较小，通信与JSON编码开销在总耗时中也占有一定比例，这说明当前系统更适合作为演示型原型，而不是以吞吐量最大化为目标的工业系统。

从结果讨论角度看，本文系统的优势主要体现在三点。第一，系统结构完整，能够把DPF这一密码学机制嵌入真实前后端流程中，而不是停留在命令行级别的单点验证。第二，共享核心经过裁剪后具备较好的可移植性，降低了实验环境依赖，使系统更适合在本科答辩环境中部署演示。第三，前端通过协议时间线和JSON面板把原本较抽象的查询过程可视化，增强了系统的可解释性。与此同时，系统也存在一定局限，例如依赖双服务器不串通假设，只支持单条精确查询，尚未进一步扩展到多关键词或更大规模数据集。这些局限并不会否定当前工作的价值，而是说明本文实现的是一个围绕核心问题构建的第一版原型。

6 总结与展望

本文围绕用户查询目标隐私保护问题，设计并实现了一个基于分布式点函数的双服务器私有查询演示系统。系统从公开商品列表与私密详情查询场景出发，构建了客户端、服务器A、服务器B和本地数据副本协同工作的完整流程，实现了目标记录映射、DPF份额生成、双路本地聚合求值、客户端结果重构以及协议过程可视化展示。为了保证系统在普通环境中的可运行性，本文对共享DPF核心进行了可移植化裁剪，形成了能够支撑后端集成和前端接口调用的核心模块。测试结果表明，系统能够稳定完成单条记录私有查询，并在版本不一致、请求失败和重构异常等情况下给出明确反馈。

从毕业设计角度看，本文工作的意义不仅在于给出了一套双服务器私有查询方案，更在于证明了该方案可以通过较为清晰的工程结构落地为一个完整的演示系统。系统同时具备理论基础、实现路径和页面展示，能够较好地支撑论文写作与答辩演示。未来工作可以从三个方向继续推进：其一，引入真正的Wasm共享核心构建产物，使浏览器端与原生端进一步统一；其二，扩展查询能力，尝试支持多关键词或小范围组合查询；其三，面向更强安全模型加入结果正确性校验与恶意行为防护机制，以提升系统在更复杂应用场景中的适用性。
