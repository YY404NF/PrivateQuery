摘要
随着互联网平台、移动终端和在线服务的持续普及，用户越来越频繁地通过网络完成商品浏览、信息检索、服务预约与内容访问等操作。在这一过程中，真正敏感的内容往往不只是返回结果本身，更是“用户究竟查询了哪一条记录”这一目标信息。传统查询系统通常要求服务器直接接收记录编号、关键词或过滤条件，然后再返回对应结果，这种处理方式虽然简单高效，却会使服务器掌握用户的访问目标，并进一步暴露兴趣偏好、消费倾向甚至健康状态等隐私内容。因此，如何在保证查询可用性的同时隐藏用户真实查询目标，已经成为隐私计算与安全系统研究中的重要问题。
本文对双服务器私有查询机制及其系统化实现进行了研究，提出了一种面向公开商品列表与私有查询场景的双服务器查询方案。该方案以分布式点函数为核心，通过将目标索引拆分为两份查询份额，分别发送至服务器A和服务器B，使任意单个服务器只能看到局部信息，无法独立判断用户实际选择的记录。
本文实现了一个由前端客户端、双服务器后端和共享DPF核心组成的原型系统。系统前端采用Vue 3技术栈，后端采用Go、Gin和SQLite实现统一服务框架，并通过C接口调用C++20实现的共享DPF核心完成本地聚合求值。测试结果表明，系统能够正确完成目录加载、版本校验、目标记录查询、结果重构和异常提示等完整流程。本文工作说明，基于分布式点函数构建双服务器私有查询系统不仅具有明确的理论依据，也具备较好的工程实现性和展示价值。
关键词：分布式点函数；双服务器；私有查询；隐私保护；前后端分离
ABSTRACT
With the continued growth of online platforms, mobile terminals, and online services, users increasingly complete product browsing, information retrieval, service reservation, and content access through networked systems. In this process, what is truly sensitive is often not only the returned result itself, but also the query target, namely which exact record the user chooses to access. Traditional query systems usually require the server to directly receive record identifiers, keywords, or filter conditions before returning results. Although this design is simple and efficient, it also reveals the user’s access target and may further expose private information such as preferences, consumption tendencies, and health-related interests. Therefore, how to hide the real query target while preserving query usability has become an important issue in privacy computing and secure system research.
This paper studies the two-server private query mechanism and its systematic implementation, and proposes a two-server query scheme for the scenario of a public product catalog with private query. Built on Distributed Point Functions, the scheme splits the target index into two query shares and sends them separately to Server A and Server B, so that any single server can observe only partial information and cannot independently determine the record selected by the user.
This paper implements a prototype system composed of a frontend client, two backend servers, and a shared DPF core. The frontend adopts Vue 3, while the backend uses Go, Gin, and SQLite and invokes a shared DPF core implemented in C++20 through a C interface to complete local aggregation. Experimental results show that the system can correctly complete catalog loading, version checking, target record querying, result reconstruction, and exception reporting. These results indicate that a two-server private query system based on Distributed Point Functions has clear theoretical support and solid engineering feasibility.
KEY WORDS:Distributed Point Function; Two-Server Model; Private Query; Privacy Protection; Frontend and Backend Separation

第一章绪论
1.1课题背景和意义
数字化服务的普及显著改变了用户访问数据的方式。无论是在线购物、电子政务、在线教育还是医疗咨询，用户越来越习惯通过统一界面浏览公开目录，再针对某一条记录发起进一步查询。从业务角度看，这是一种非常常见的交互形式；但从隐私保护角度看，用户点击了哪一条记录、对哪一项内容进行了深入查看，往往比最终返回的页面内容更具敏感性。尤其当查询对象与个人健康、金融偏好、研究兴趣或采购意图相关时，目标索引本身就可能成为可被分析和利用的隐私信息。
传统查询系统通常按照“客户端提交目标参数，服务器执行检索并返回结果”的方式工作。该模式在实现上清晰直接，但也使服务器拥有用户查询目标的可见性。近年来，国内关于隐私计算的研究不断扩展，既关注联邦学习中的隐私泄露与防护问题，也关注数据库加密、可搜索加密和可信查询机制等方向。汤凌韬等对联邦学习中的隐私攻击、模型反演、成员推断和梯度泄露等问题进行了系统梳理，指出隐私保护不能只停留在数据存储阶段，而应覆盖计算与访问全过程[1]；王腾等进一步总结了联邦学习场景下的隐私保护技术路径，说明访问模式与交互过程同样可能形成隐私风险[2]。这些研究虽然不直接等同于私有查询协议，但共同强调了一个事实：在现代数据服务中，仅仅保护数据内容并不足够，访问行为本身也需要被纳入保护范围。
与此对应，国外在私有信息检索、函数秘密共享和分布式点函数方面已经形成较为连续的研究脉络。Boyle等在函数秘密共享研究中系统给出了点函数份额化的核心思想，为后续以点查询为中心的协议设计奠定了理论基础[3]。随后，围绕PIR与DPF的研究开始向认证性、低通信、批处理和多参与方扩展：Colombo等提出的Authenticated Private Information Retrieval在隐藏查询目标的同时强化了结果认证能力[4]；de Castro等提出的WhisPIR则进一步压缩通信量并减少预处理依赖[5]。可以看到，国际研究已不再满足于“能不能隐藏目标”，而更强调在不同约束下如何优化协议结构与系统效率。
从系统研究角度看，关键在于如何将成熟的理论机制转化为能够运行、能够验证且具有清晰业务语义的完整系统。本文选择“公开商品列表与私有查询”这一场景，正是因为它兼具直观交互流程与明确隐私动机：公开商品名称、类别、价格等信息可以直接展示，但用户点开哪一件商品的详情应尽量不被单个服务器掌握。如图1-1所示，传统详情查询直接把目标编号发送给服务器，而双服务器私有查询则先在客户端生成两份查询份额，再由两台服务器分别求值，最后由客户端重构结果。两种流程在用户界面上看似接近，但在隐私泄露面上存在本质区别。
图1-1 传统查询与双服务器私有查询对比
基于上述背景，本文研究的意义主要体现在三个方面。其一是理论意义。通过把分布式点函数、双服务器模型和结果重构过程嵌入实际查询场景，可以使抽象的密码学概念获得更清晰的系统语境。其二是工程意义。相比停留在算法论文或命令行测试层面，设计一套包含前端、后端、数据库和共享核心的完整系统，更能检验方案的可落地性。其三是应用意义。带有协议时间线、JSON过程展示和查询页面联动的实现方式，有助于更直观地说明私有查询的工作机理，并验证系统设计与协议流程之间的一致性。
1.2国内外研究现状
围绕“在不暴露真实查询目标的前提下获取结果”这一问题，国内外研究形成了不同但互相关联的发展路径。国外研究的主线更接近密码学协议本身，重点关注私有信息检索、函数秘密共享和分布式点函数等基础机制；国内研究则更多从隐私计算、数据库安全、可搜索加密和应用场景扩展等角度切入，强调安全能力如何与现有系统结合。
从国外研究来看，函数秘密共享与DPF是双服务器私有查询的重要基础。Boyle等提出的Function Secret Sharing说明了如何将特定函数拆分为多个份额，并在多方协同下恢复目标函数值，为点函数的份额化表达提供了经典框架[3]。在此基础上，Colombo等提出的Authenticated Private Information Retrieval引入认证机制，使私有查询不只关注目标隐藏，还同时考虑结果的可信性[4]；de Castro等提出的WhisPIR关注无状态和低通信量场景，说明近期PIR研究正在向更轻量的部署条件推进[5]。与此同时，Piske等研究了批处理DPF构造，尝试降低多目标或批量查询时的份额开销[6]；Cheng等将增量式DPF用于两方安全聚合，说明DPF已不再局限于传统PIR语境，而是在更广泛的隐私协议中发挥作用[7]；Goel等则进一步研究了多方DPF结构，展示了从双参与方到多参与方扩展的可能性[8]。整体上看，国外研究已经从“协议是否成立”逐步转向“协议如何更高效、更易认证、可否扩展到更复杂场景”的阶段。
国内近三年的研究重点则更偏向隐私保护技术与应用结合。汤凌韬等和王腾等分别从联邦学习隐私问题与隐私保护技术两个角度进行了系统综述，指出当前国内对隐私泄露问题的关注已从原始数据保护延伸到训练过程、模型参数和协同计算过程[1-2]。虽然这类工作并未直接针对双服务器私有查询，但其提出的“访问轨迹与交互行为同样可能泄露隐私”的观点，对本文选题具有重要启发。李旭东等面向SQLite提出多密钥页级别加密系统，说明国内在数据库安全和轻量部署方向已经积累了一定工程经验[9]。在可搜索加密领域，闫玺玺等提出基于区块链的多关键词模糊搜索加密方案，尝试提升检索表达能力[10]；李宇溪等提出基于区块链索引的可验证布尔搜索加密方案，在可验证性与索引组织方面进行了进一步探索[11]。这些研究表明，国内已经较多关注“如何在数据加密前提下完成检索”“如何增强结果可验证性”“如何与区块链或数据库环境结合”等问题。
1.3本文研究内容
围绕双服务器私有查询系统的构建目标，本文主要完成了三个方面的工作。第一，结合公开商品列表与私有查询场景，对系统角色、查询对象和隐私边界进行了明确建模，确定系统需要保护的是用户选择的目标记录索引，而不是公开展示的目录信息。第二，围绕分布式点函数的实际应用需求，设计了记录定长编码方案、双服务器本地聚合求值流程以及客户端结果重构机制，使DPF输出能够与业务数据结构稳定衔接。第三，基于Vue 3、Go、Gin、SQLite和C++20实现了完整原型系统，并通过构建验证、原生测试与功能测试对系统正确性和稳定性进行检查。
第二章私有查询相关基础与实现技术
2.1私有查询与双服务器模型
私有查询的核心目标，是让用户在获得所需数据的同时，尽量不向服务提供者暴露真实查询目标。与传统意义上的传输加密或数据加密不同，私有查询关注的是访问行为本身。如果服务器能够直接看到用户请求的记录编号、关键词或者目标位置，即便数据在传输链路上经过加密，查询意图仍然没有被隐藏。换言之，私有查询保护的重点是“查了什么”，而不是“传输内容有没有被窃听”。
双服务器模型是私有查询中较常见、也较适合系统原型实现的一种结构。如图2-1所示，客户端面对的是两个保存同一版本数据副本的服务器。用户选择某个目标记录后，客户端不会把真实索引直接发送给任意一侧，而是先把该索引转换为两份查询份额。服务器A与服务器B分别根据各自收到的份额在本地数据副本上求值，随后将结果份额返回给客户端。只有当两份结果在客户端侧合并时，目标记录才会被恢复出来。由此，任意单个服务器都不足以独立判断真实查询目标。

2.3双服务器模型的安全假设与适用边界
任何私有查询系统都不可能脱离安全假设独立成立。本文系统之所以能够隐藏查询目标，最关键的前提是服务器A与服务器B在协议执行过程中不发生串通。如果两台服务器交换各自收到的份额或协同分析中间结果，真实索引就可能被恢复出来。与此同时，本文系统默认服务器属于“诚实但好奇”模型，即服务器会按协议执行本地求值，但可能尝试从自身份额、局部输出和请求上下文中推测用户意图。

2.4系统实现相关技术
本文系统的实现遵循“结构清晰、依赖适中、便于部署和说明”的原则。后端采用Go语言与Gin框架实现统一服务代码。Go语言适合组织清晰的接口层、服务层和数据访问层，Gin则能够快速构建HTTP路由和JSON接口。数据库选择SQLite，主要考虑到其部署简洁、文件化管理方便，能够较好适应本地化运行和轻量部署需求。根据仓库中的实现，后端默认通过HOST、PORT、PARTY、DB_PATH和WORKER_COUNT等配置区分服务器身份与运行参数。
前端部分采用Vue 3、Vite、Vue Router与Axios实现。Vue 3便于以组件化方式组织首页、演示页、商品卡片、详情卡片和协议时间线等界面，Vite则用于支撑快速开发与生产构建。路由层中，系统保留/和/demo两个主要页面，并将/process与/scene重定向到/demo，使查询演示页面成为统一的核心入口。Axios负责同时向服务器A与服务器B发起请求，而状态层负责保存版本信息、活动商品、查询阶段和最近一次完整协议过程。
共享DPF核心采用C++20实现，并通过稳定的C接口提供给Go后端调用。该核心负责密钥份额生成所需的底层数据结构表达、定义域求值与聚合计算能力，同时将输出群固定为uint64模2的64次方加法群，以便与记录块编码保持一致。前端侧则调用与共享核心一致的查询生成与重构逻辑，从而使浏览器端的份额生成、服务器端的本地求值以及客户端的结果恢复处于统一的协议框架中。
第三章系统需求分析与总体设计
3.1应用场景与需求分析
本文系统以“公开商品列表与私有查询”为具体应用场景。用户进入系统后，可以直接看到公开商品列表，包括商品名称、类别、价格、商家名称和更新时间等信息。这部分内容本身并不需要被隐藏，因为它承担的是场景入口与目录导航作用。真正需要保护的是用户在目录中点开了哪一件商品、希望查看哪一条详情记录。如果系统直接把记录编号发送给服务器，那么服务器就会立即得知用户的真实选择，隐私保护目标也就无法成立。
结合仓库中的前后端实现，系统的核心参与方可以划分为四类：前端客户端、服务器A、服务器B以及双侧数据副本。如图3-1所示，前端不仅承担普通页面渲染任务，还负责目标映射、份额生成、结果重构和过程展示；服务器A与服务器B承担逻辑对称的本地聚合求值任务；双侧SQLite副本负责提供一致的数据基础。这样设计的原因在于，客户端天然更适合作为“查询发起者与结果恢复者”，而服务器只保留各自局部可见的信息。
图3-1系统总体架构图
在这一场景下，系统需要同时满足两类要求。一类是隐私性要求，即单个服务器不能直接获得用户目标索引；另一类是可用性要求，即用户仍然能够完成“浏览目录、点击详情、查看结果”的自然交互。如果只强调协议安全而忽略用户界面和系统流程，那么该方案难以形成完整系统；如果只强调页面效果而没有可靠的查询隐藏机制，那么系统又无法体现课题的研究价值。因此，需求分析必须同时覆盖密码协议、数据组织、页面交互与异常处理四个层面。

3.4 系统总体流程设计
在整体流程上，客户端既是查询发起者，也是结果重构者。用户进入演示页后，前端首先并发读取服务器A和服务器B的版本信息与公开目录；若双侧版本不一致，则系统停止后续查询。版本一致时，用户点击某一条商品记录，前端根据该记录的recordId生成两份DPF查询份额，并分别提交给两台服务器。服务器收到份额后，从本地数据库读取完整载荷块数组，调用共享核心执行聚合求值，返回各自的结果份额。最后，客户端逐块执行模2的64次方加法，并按固定布局解码为详情对象。图3-4展示了这一流程中的时序关系。
图3-4客户端查询时序图
这一流程的一个重要特点是职责分离。服务器A和服务器B都不会单独恢复最终详情，客户端也不会直接接触数据库原始块数组。通过这种方式，系统能够同时满足“单侧不可见真实目标”和“客户端可恢复完整业务结果”两个目标。另外，前端将最近一次完整查询过程保存在状态对象中，因此协议时间线可以同步展示生成输入、请求报文、响应报文和重构结果，这增强了系统的可解释性与过程可观测性。
第四章系统关键模块设计与实现
4.1 共享DPF核心设计与实现
共享DPF核心是本文系统中最接近协议层的位置。该模块采用C++20实现，主要负责点查询份额表达、定义域求值与载荷聚合计算，并通过C接口对外提供稳定调用方式。对本文系统而言，共享核心的意义并不只是“把算法跑起来”，更重要的是把前端生成的查询份额、后端读取的载荷块以及客户端重构的结果统一到同一套数据表示中。
如图4-1所示，共享核心内部可以划分为四个主要部分：Block128等底层数据结构，用于表达种子和纠正字；KeyShare相关结构，用于保存一侧查询份额；全域求值与聚合能力，用于在定义域上生成选择结果并对载荷块求和；C接口封装，用于被Go后端调用。这种模块划分的优势在于：既保留了算法表达所需的数据结构，又避免让上层服务直接依赖复杂的内部实现细节。

4.2 后端服务设计与实现
后端部分采用统一代码库实现服务器A与服务器B，两者通过运行配置区分身份。根据config.Load()的默认设置，服务器A使用PORT=8081、PARTY=0、数据库文件server_a.db；服务器B使用PORT=8082、PARTY=1、数据库文件server_b.db。系统启动时，会先打开SQLite数据库、检查表结构、写入样例记录与版本元数据，然后再启动Gin路由服务。这种“统一代码 + 配置区分”的方式能够清楚体现双服务器结构中的逻辑对称性。
在请求处理流程上，后端主要提供四类接口：健康检查、版本信息、公开目录和查询求值。其中，/api/query/eval是最关键的接口，它负责接收一侧查询份额、执行版本与参数校验、读取载荷块数组、调用共享核心聚合求值，并将结果份额以十六进制字符串数组形式返回。图4-2展示了该接口的处理流程。

4.5 异常处理与版本一致性设计
要保证系统行为完整，仅有成功路径是不够的，还必须对失败路径进行明确设计。本文系统首先在初始化阶段读取双服务器版本信息，并在前端状态层设置versionMismatch标记。一旦发现两侧数据版本不同，系统就不再继续发起私有查询，而是直接给出提示。这种设计的意义在于，双服务器私有查询依赖双侧副本的一致性；如果版本不一致，即便密码协议本身没有问题，重构结果也可能出现偏差。
除了版本问题之外，系统还对份额格式、定义域大小和请求异常进行了区分处理。后端在/api/query/eval中对datasetVersion、domainSize以及keyShare结构进行检查，并根据错误类型返回不同状态码；前端则保留最近一次完整查询过程，即使请求失败，也不会立刻丢弃生成输入和已发出的请求体。表4-2汇总了主要异常场景与处理方式。保留这些异常处理逻辑，有助于说明系统不仅能够完成主流程，也考虑了边界情况。

5.3 性能分析与结果讨论
本文的性能分析并不把极限吞吐量作为主要衡量点，而是关注系统在既定实验条件下能否保持稳定的查询体验。样例数据集中共有32条记录，每条记录被编码为20个块。一次查询执行时，服务器端需要读取载荷块并完成份额求值，客户端则负责生成查询份额、协调双路请求，并在收到响应后恢复结果。由于查询链路并不复杂，服务器A和服务器B又可以并行处理请求，所以系统能够较快完成一次完整查询，页面反馈也比较自然。
后端响应中保留了elapsedMs等耗时字段，同时还会返回工作线程数、记录数和块数；前端状态层也记录了查询开始、结束以及阶段变化。这些信息虽然不等同于完整的性能压测报告，但已经为后续观察系统运行状态提供了基础。通过多次重复查询，可以分别记录份额生成、双路请求和结果重构的耗时，再结合页面状态判断系统响应是否稳定。这样处理的好处是，性能讨论没有停留在抽象描述上，而是能够回到系统实际输出的观测字段中。
整体运行结果说明，本文系统已经能够把分布式点函数嵌入真实前后端流程，并把原本较抽象的查询过程呈现在页面中。后端通过统一代码区分双服务器身份，前端通过状态管理串联目标选择到结果展示的过程，固定块编码也让密码计算和业务数据之间保持了较清晰的对应关系。当然，系统当前仍以单条精确查询为主，对于更复杂的检索方式和更强的安全需求，仍需要在后续工作中继续展开。
第六章总结与展望
6.1 工作总结
本文围绕公开目录场景下的查询目标隐藏问题，设计并实现了一个基于分布式点函数的双服务器私有查询系统。传统详情查询会直接暴露记录编号，而本文采用双服务器模型和查询份额机制，把目标索引拆分为两份局部信息后分别交给服务器A和服务器B处理。这样一来，单个服务器只能看到自己收到的份额，难以独立恢复用户真实选择。为了让这一机制能够落到具体场景中，系统先将商品记录编码为固定长度块，再由两台服务器分别完成本地聚合，最后由客户端恢复查询结果。
具体实现时，共享DPF核心负责提供底层求值能力，Go后端负责接口处理和数据读取，前端则承担用户交互与协议过程展示。用户可以先浏览公开商品列表，再点击某一商品发起私有查询；客户端生成查询份额并发送双路请求，双服务器返回结果份额后，再由客户端恢复结构化详情对象。测试与构建验证表明，系统能够完成查询前的数据加载，也能够在版本校验、结果重构和异常提示等环节保持稳定表现，说明本文方案在工程实现和系统完整性方面具有一定可行性。
6.2 工作展望
后续工作可以在现有系统基础上继续向更实用的查询形态扩展。目前系统主要支持单条精确查询，已经能够说明双服务器私有查询的基本过程，但距离更复杂的真实检索场景仍有一定距离。后续可以先尝试支持带有多个条件的查询，再逐步考虑批量请求，使系统能够处理更丰富的用户需求。与此同时，当前安全模型主要建立在双服务器不串通和诚实但好奇假设之上，如果要面向更严格的应用环境，还需要进一步考虑服务器返回结果是否可信，以及多次查询之间是否可能产生关联。除此之外，系统也可以继续完善性能统计和实验数据采集方式，让后续优化有更稳定的观测依据。
